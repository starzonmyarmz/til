A payouts report shipped with every value hardcoded into JSX — `$213,596.57`, `3/5/2026`, the works. Replacing those placeholders with values pulled from a JSON payload is supposed to be the boring step. It only stays boring if the rendering side commits to a few habits about how money and dates cross the wire.

What follows are three of them. None of them is individually exciting. Together they are the difference between a table that renders correctly in every locale and one that ships a subtle bug to whichever customer happens to live in the wrong timezone.

## 1. Integer minor units on the wire, currency on the surface

The payout amount is an integer. It is always an integer — `21359657` minor units, never `213596.57` dollars. Sending a float across the wire opens the door to every binary-floating-point story that finance teams already know by heart, and shipping a pre-formatted string would hand the locale decision to the server in a place where it does not belong.

So the server sends an integer and the client decides what to show.

```tsx
const currencyFormatter = new Intl.NumberFormat('en-US', {
  style: 'currency',
  currency: 'USD',
})

// ...
<h2 className="title-lg m-0">
  {currencyFormatter.format(payout_amount / 100)}
</h2>
```

`payout_amount / 100` is the one place dollars exist in the whole pipeline, and it lives inside the formatter call where the unit converts back to its display form. The integer never leaks. The display string is never compared, summed, or persisted.

The same number, four different locales, four different conventions — symbol position, decimal separator, grouping separator, and whether minor units are shown at all. The server does not need to know any of that. It only needs to know how many minor units the bank actually moved.

A subtle bug avoided: the fee row negates the value before formatting, not after.

```tsx
{currencyFormatter.format(-payout_fee / 100)}
```

`-payout_fee / 100` is a number; `'-' + formatter.format(payout_fee / 100)` is a string with a leading hyphen-minus that the locale’s own negative convention may or may not match. Letting `Intl` see a negative number lets it produce the right thing — accounting parentheses, locale-specific minus glyphs, the works — instead of a hand-rolled approximation.

The principle is the same one that motivates parameterized SQL. Hand the formatter structured data, not a half-assembled string. The library exists to make that conversion correct.

## 2. Hoist the formatter

`Intl.NumberFormat` is more expensive to construct than most React code assumes. Every `new Intl.NumberFormat(locale, options)` call walks the locale data, resolves the options, and allocates an internal pattern. None of that work needs to happen more than once for a fixed `(locale, options)` pair, and on a table with hundreds of rows it adds up fast.

The fix is one line:

```tsx
const currencyFormatter = new Intl.NumberFormat('en-US', {
  style: 'currency',
  currency: 'USD',
})

export default function PayoutReports({ ... }) { ... }
```

Module scope, top of file, built exactly once for the lifetime of the page. Every row of the transaction table reuses the same instance. The same applies to `Intl.DateTimeFormat`, `Intl.Collator`, `Intl.PluralRules` — any of the `Intl.*` constructors.

<FormatterScopeDemo client:load />

The numbers vary by machine and browser, but the ratio rarely surprises anyone — re-constructing the formatter inside the loop costs an order of magnitude or two more than reusing the hoisted instance. A two-hundred-row payouts table calling `new Intl.NumberFormat()` for every cell would have spent more time configuring formatters than rendering DOM. None of that effort is visible in a code review — which is exactly why the habit needs to be the default.

A related habit worth naming: do not put `useMemo(() => new Intl.NumberFormat(...))` inside the component. The component is the wrong scope. Currency formatters are pure functions of `(locale, options)`; they do not depend on props or state. Putting them in module scope says that out loud and removes the need for a memo dependency array that nobody will ever update correctly.

If the locale really does change at runtime — a user toggling regions, an admin previewing another tenant — the right move is still a small map of pre-built formatters keyed by locale, not on-demand construction inside the render path.

```tsx
const FORMATTERS: Record<string, Intl.NumberFormat> = {
  'en-US': new Intl.NumberFormat('en-US', { style: 'currency', currency: 'USD' }),
  'en-GB': new Intl.NumberFormat('en-GB', { style: 'currency', currency: 'GBP' }),
  'de-DE': new Intl.NumberFormat('de-DE', { style: 'currency', currency: 'EUR' }),
}
```

That table is still allocated once per module load, and the lookup is `O(1)`. The render path stays a single property access.

## 3. ISO strings in, locale-aware dates out

Dates have the same shape of bug as money. The server has a `Date` (or `Time`) object. The wire format is a string. The client parses the string and the renderer chooses how to show it.

The boring choice — and the right one — is ISO 8601 with an offset:

```ruby
transaction_date: bt.transaction_date&.iso8601,
```

```tsx
const dateFormatter = new Intl.DateTimeFormat('en-US')

// ...
{transaction.transaction_date
  ? dateFormatter.format(new Date(transaction.transaction_date))
  : ''}
```

`new Date(string)` is one of the rare places where `Date` parsing is actually well-defined: ISO 8601 with an offset is required by the spec. Locale-formatted strings, ambiguous two-digit years, and the bare-date form `"2026-03-05"` — which is parsed as UTC midnight, then displayed in local time, which has tripped every team that ever wrote a calendar — are the failure modes the ISO format dodges.

The same instant rendered into four locales. Notice that the parsed value never changes — it is a fixed point in time. Only the formatter output changes. That is the whole separation: time is a number, display is a render concern, and the wire format is the contract between them.

A few small things worth being deliberate about:

- **Always include the offset.** `2026-03-05T23:30:00-05:00` is unambiguous. `2026-03-05T23:30:00` is not — different browsers have historically disagreed about whether the absence of an offset means UTC or local time. The spec eventually settled on local, but code written under either reading still exists in the wild.
- **Check the parse before formatting.** `new Date("not a date")` returns an `Invalid Date` whose `.getTime()` is `NaN`, and `Intl.DateTimeFormat` will happily render the string `"Invalid Date"` straight into the DOM. A `Number.isNaN(date.getTime())` guard catches it.
- **Render the nullable case explicitly.** `transaction.transaction_date ? ... : ''` looks like noise until the first row arrives without a date and the table throws `Invalid Date` at every QA tester. The null branch is part of the contract, not an edge case.
- **Reach for `dateStyle` and `timeStyle` before `format`-with-options.** `{ dateStyle: 'medium' }` is more locale-correct than hand-picking `year`, `month`, and `day` — it lets the locale data decide whether a comma belongs between the month and the day, whether the year leads or trails, and so on. The hand-picked option set is only worth reaching for when a specific column header constraint demands it.

## What this adds up to

The three habits reinforce each other. Integer minor units are only worth keeping integer if the formatter can produce the right currency string at the edge — that is what `Intl.NumberFormat` does. The formatter is only cheap to use everywhere if it is hoisted out of the render path — that is what module scope does. And neither habit matters if the date column is silently rendering UTC midnight as the day before because someone shipped a bare date string — that is what ISO offsets and a `Number.isNaN` check fix.

The whole point of `Intl` is to compress a pile of locale knowledge into a few well-named primitives. The cost of that compression is that the primitives are not free, and the wire format has to meet them halfway. Spend the boring effort once and the second table like this one falls out of the keyboard in an afternoon.
